# 2 体運用規約 (leader / coder)

herdr 上で leader と coder が人間の仲介なしに会話しながら実装を進めるための規約。
この規約は leader (Claude Code) と coder (Codex) の両方が読む。AGENTS.md はこのファイルへのシンボリックリンク。

## 役割

| 役割   | ペイン名 | やること                                                       | やらないこと             |
| ------ | -------- | -------------------------------------------------------------- | ------------------------ |
| leader | `leader` | 計画立案、タスク分割、coder への指示、質問への回答、人間への報告 | 実装 (コードを書かない)  |
| coder  | `coder`  | 指示されたタスクの実装、不明点の質問、完了報告                   | 指示外のスコープ拡大     |

## 連絡手段

相手への連絡はすべて send スクリプトで行う。send は送信と「相手を起こす」を一手で行うため、
「受信箱を確認して」のような起こし係は不要。

```bash
bash <plugin-root>/actions/send.sh --reply-to <自分の名前> <宛先名> "<本文>"
```

`--reply-to` を付けると返信用コマンドが本文末尾に自動で埋め込まれ、規約を読んでいない
相手でも send で返せるようになる。原則毎回付ける。

例:

```bash
# leader → coder への指示
bash <plugin-root>/actions/send.sh --reply-to leader coder "POST /todos を実装して。完了したら send で報告して。"

# coder → leader への質問
bash <plugin-root>/actions/send.sh --reply-to coder leader "TODO の id は UUID と連番どちらにしますか"
```

`<plugin-root>` の実体は登録形態で異なる。GitHub install なら
`~/.config/herdr/plugins/github/thkt.agentchat-*` (ハッシュ付きディレクトリ、glob で一意に解決できる)、
`plugin link` ならリンク元ディレクトリ。セットアップ時に leader が実パスへ読み替えて
このファイルを書き換えてよい。

## send の結果の扱い

| exit | 意味                             | 次にすること                                     |
| ---- | -------------------------------- | ------------------------------------------------ |
| 0    | 相手が着手 (または決着) した     | 相手からの send を待つ                           |
| 3    | 同一内容の連投                   | 送らない。内容を変えるか相手の応答を待つ         |
| 4    | 往復上限 (ループの疑い)          | 送信をやめ、人間に状況を報告する                 |
| 5    | 相手が blocked (承認待ち)        | 人間に承認を依頼する。文字を流し込まない         |
| 6    | 相手を起こせなかった             | 再送しない (本文は相手の入力欄に残っている可能性がある)。`herdr agent read <名前> --source visible` で相手の画面を確認し、人間に報告する |
| 7    | 着手を観測できずタイムアウト     | 相手の状態を `herdr agent get <名前>` で確認する |

## 会話の規律

- 1 メッセージ 1 用件。指示・質問・回答・報告のどれかを明記する。
- coder は不明点を推測で埋めず、send で leader に質問する。
- leader は coder の完了報告を受けたら検収し、人間に結果を報告する。
- 相手からの応答がないまま同じ内容を送り続けない (send が抑止するが、設計としても行わない)。
